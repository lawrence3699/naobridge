# 🧠 "脑桥" — 脑损伤患者交流论坛小程序 · 项目总规划

> **立项宗旨**：以人道主义和理性主义为基石，为脑损伤患者及其家属搭建一个安全、温暖、无障碍的线上交流社区。
>
> **产品形态**：微信小程序
>
> **文档版本**：v1.0 | 2026-03-11

---

## 一、项目愿景与核心原则

### 1.1 我们要解决的问题

脑损伤（含脑外伤、脑卒中、脑瘫等）患者群体在日常生活中面临严重的社交孤立。他们需要：

- 一个**不被歧视**的空间，自由表达康复中的困惑与进展
- 与**同路人**交流的渠道，获取真实的经验参考
- 了解**脑机接口等前沿技术**的窗口，保持对未来的希望
- 家属之间**互相支持**的平台，缓解照护压力

### 1.2 三条铁律（贯穿产品设计始终）

| 原则 | 具体含义 |
|------|----------|
| 🕊️ **人道主义优先** | 所有功能设计以用户的尊严和安全感为第一优先级，零容忍任何形式的歧视、嘲讽、虚假医疗信息 |
| 🧩 **无障碍为默认** | 不是"加一个无障碍模式"，而是默认交互就是简单、清晰、大字体、高对比度的 |
| 🔬 **理性主义护航** | 鼓励科学讨论，标记未经验证的信息，防止伪科学和虚假疗法传播 |

## 二、功能清单与优先级

### 2.1 MVP（最小可用版本）— 第一期必做

#### ✅ 用户注册与登录
- 微信一键授权登录（无需额外注册流程，降低使用门槛）
- 首次登录引导填写基本资料：
  - 昵称（支持匿名，保护隐私）
  - 身份标签：患者本人 / 家属 / 关注者（仅自选，不强制）
  - 头像（可用系统默认头像）
- 用户协议 & 社区公约（必须同意后方可使用）

#### ✅ 发帖功能
- 支持**图片+文字**混排帖子
- 图片上传：单帖最多 9 张图，自动压缩
- 文字：最少 10 字，最多 5000 字
- 帖子分类标签（发帖时必选一个）：
  - 💬 康复日记
  - 🔬 脑机接口讨论
  - ❤️ 情感互助
  - 📚 知识科普
  - ❓ 求助问答
  - 🗣️ 自由话题
- 发帖前自动敏感词过滤（屏蔽广告、诈骗、极端言论关键词）

#### ✅ 评论功能
- 支持对帖子进行文字评论
- 支持评论的**楼中楼**回复（二级评论）
- 评论同样经过敏感词过滤
- 帖主可选择关闭评论

#### ✅ 举报功能
- 举报对象：帖子 / 评论 / 用户
- 举报类型（预设选项）：
  - 虚假医疗信息
  - 广告 / 诈骗
  - 人身攻击 / 歧视
  - 色情 / 暴力
  - 其他（需填写说明）
- 举报后 24 小时内处理反馈
- 举报人信息严格保密

#### ✅ 管理功能
- **管理员后台**（独立管理端小程序或 Web 后台）：
  - 帖子审核：查看被举报内容，一键删除/通过
  - 用户管理：禁言（7天/30天/永久）、封禁账号
  - 数据看板：日活、帖子数、举报数等基础统计
  - 敏感词库维护：添加/删除敏感词
- **管理员标识**：管理员在社区内发言时显示特殊角色标记

### 2.2 第二期增强功能

| 功能 | 说明 |
|------|------|
| 🔍 搜索 | 按关键词、标签、用户搜索帖子 |
| ❤️ 点赞/收藏 | 对帖子点赞和收藏，方便日后查看 |
| 🔔 消息通知 | 评论提醒、举报处理结果通知、系统公告 |
| 👤 个人主页 | 查看自己的发帖/评论/收藏记录 |
| 📌 置顶/精华帖 | 管理员可置顶重要帖子，标记优质内容为精华 |
| 🏷️ 用户等级 | 根据发帖/评论活跃度给予等级徽章，鼓励参与 |

### 2.3 第三期远期功能

| 功能 | 说明 |
|------|------|
| 🎙️ 语音帖 | 为打字困难的患者提供语音发帖+语音转文字 |
| 🤖 AI 内容审核 | 接入 AI 模型辅助识别虚假医疗信息 |
| 📊 康复打卡 | 每日康复训练记录与进度追踪 |
| 🏥 专家认证 | 引入医疗专业人士认证体系，权威回答标记 |
| 🌐 无障碍增强 | 屏幕朗读适配、语音交互、超大字体模式切换 |

## 三、无障碍设计专项

> 这不是可选项，是核心需求。

### 3.1 默认设计标准

- **字体**：正文最小 16px，标题 20px+，行距 1.8 倍
- **色彩**：对比度 ≥ 4.5:1（WCAG AA 级），避免纯红/绿区分信息
- **触摸区域**：所有可点击元素最小 44×44px
- **操作简化**：核心流程（发帖、评论、举报）不超过 3 步
- **反馈明确**：所有操作有清晰的成功/失败提示，使用文字而非仅颜色

### 3.2 渐进增强（第二/三期）

- 字体大小可调（小 / 中 / 大 / 超大）
- 高对比度模式开关
- 语音输入替代打字
- 简洁模式（隐藏非核心功能，只保留发帖/看帖/评论）

## 四、技术方案（面向非技术人员的白话说明）

### 4.1 整体架构

```
┌─────────────────────────────────────────┐
│            用户的微信                      │
│         ┌──────────────┐                 │
│         │  小程序前端   │                 │
│         │ (用户看到的)  │                 │
│         └──────┬───────┘                 │
└────────────────┼─────────────────────────┘
                 │ 网络请求
                 ▼
┌─────────────────────────────────────────┐
│          微信云开发 (后端)                 │
│  ┌──────────┐ ┌──────────┐ ┌─────────┐  │
│  │ 云函数    │ │ 云数据库  │ │ 云存储   │  │
│  │ (业务逻辑)│ │ (数据)   │ │ (图片)  │  │
│  └──────────┘ └──────────┘ └─────────┘  │
└─────────────────────────────────────────┘
```

### 4.2 为什么选微信云开发？

这是为非技术团队量身推荐的方案：

| 优势 | 说明 |
|------|------|
| **零运维** | 不用自己买服务器、配域名、搞备案，腾讯云全托管 |
| **成本低** | 基础版免费，日活千级用户月成本约 50-200 元 |
| **安全** | 自带 DDoS 防护、数据加密，符合微信安全规范 |
| **快速上线** | 省去 50% 以上的基础设施工作量 |

### 4.3 数据库核心表设计（简化版）

| 数据表 | 存储内容 | 关键字段 |
|--------|----------|----------|
| `users` | 用户信息 | openid、昵称、身份标签、注册时间、状态（正常/禁言/封禁） |
| `posts` | 帖子 | 标题、正文、图片URL列表、分类标签、作者ID、创建时间、状态 |
| `comments` | 评论 | 内容、帖子ID、作者ID、父评论ID（楼中楼）、创建时间 |
| `reports` | 举报记录 | 举报人、被举报对象、举报类型、处理状态、处理结果 |
| `admins` | 管理员 | openid、权限等级、操作日志 |
| `sensitive_words` | 敏感词库 | 词条、类型、添加时间 |

### 5.2 各 Agent 职责详解

#### 🎯 PM Agent（项目经理）
- **角色**：全局协调者，任务拆解与分配
- **职责**：
  - 将本规划文档拆解为可执行的开发任务（Issue）
  - 定义每个迭代的交付标准
  - 协调各 Agent 之间的接口对接
  - 跟踪进度，处理阻塞问题
- **输入**：本 plan.md 文档
- **输出**：任务看板（Trello/飞书）、迭代计划、验收清单

#### 🎨 UI Agent（前端开发）
- **角色**：小程序前端界面开发
- **技术栈**：微信小程序原生框架 / Taro（推荐）
- **职责**：
  - 根据设计稿实现所有页面（首页、帖子列表、帖子详情、发帖、个人中心）
  - 实现无障碍设计标准（大字体、高对比度、大触摸区）
  - 与 Backend Agent 对接 API
  - 适配不同屏幕尺寸
- **交付物**：可运行的小程序前端代码

#### ⚙️ Backend Agent（后端开发）
- **角色**：云函数与数据库开发
- **技术栈**：微信云开发（Node.js 云函数 + 云数据库 + 云存储）
- **职责**：
  - 实现用户注册登录、发帖、评论、举报、管理等全部 API
  - 数据库设计与索引优化
  - 图片上传与压缩处理
  - 敏感词过滤引擎
  - 管理后台接口
- **交付物**：全套云函数代码、数据库初始化脚本、API 文档

#### ♿ A11y Agent（无障碍专项）
- **角色**：无障碍体验守护者
- **职责**：
  - 审查所有页面的无障碍合规性
  - 测试屏幕阅读器兼容性
  - 验证色彩对比度、触摸区域大小
  - 模拟各类障碍场景进行操作测试
  - 提出改进建议并跟踪修复
- **交付物**：无障碍测试报告、改进建议清单

#### 📝 Content Agent（内容审核）
- **角色**：内容安全与社区氛围维护
- **职责**：
  - 编写社区公约与用户协议
  - 构建初始敏感词库（医疗诈骗、虚假疗法、歧视用语等）
  - 制定内容审核标准与流程
  - 设计举报处理 SOP（标准操作流程）
  - 编写管理员操作手册
- **交付物**：社区公约、敏感词库、审核手册、管理员指南

#### 🛡️ QA Agent（质量保障）
- **角色**：测试与质量把控
- **职责**：
  - 编写测试用例（注册流程、发帖流程、评论流程、举报流程、管理流程）
  - 执行功能测试、兼容性测试
  - 压力测试（模拟并发用户）
  - 验证数据安全与隐私保护
  - Bug 跟踪与回归测试
- **交付物**：测试用例文档、测试报告、Bug 列表

#### 🔒 Security Agent（安全合规）
- **角色**：数据安全与法律合规
- **职责**：
  - 确保符合《个人信息保护法》《网络安全法》
  - 用户数据加密方案设计
  - 隐私协议撰写
  - 微信小程序审核合规检查
  - 医疗健康类小程序的特殊资质审查
- **交付物**：合规检查清单、隐私协议、安全方案文档

### 5.3 Agent 协作流程

```
阶段一：规划期（第 1-2 周）
  PM Agent → 拆解任务 → 分配给各 Agent
  Content Agent → 输出社区公约 & 敏感词库
  Security Agent → 输出合规清单 & 资质需求

阶段二：开发期（第 3-8 周）
  UI Agent ←→ Backend Agent  (前后端联调)
  A11y Agent → 持续审查 UI Agent 输出
  Content Agent → 持续补充敏感词库

阶段三：测试期（第 9-10 周）
  QA Agent → 全面测试
  A11y Agent → 无障碍专项测试
  Security Agent → 安全渗透测试

阶段四：上线期（第 11-12 周）
  PM Agent → 协调提审
  Security Agent → 最终合规确认
  全员 → 上线支持与紧急修复
```

## 八、微信审核注意事项

医疗健康类小程序审核较为严格，需提前准备：

| 审核要点 | 应对措施 |
|----------|----------|
| **类目选择** | 建议选"社交 > 社区/论坛"而非医疗类，降低资质要求 |
| **内容安全** | 必须接入微信内容安全 API（`security.msgSecCheck`） |
| **隐私合规** | 完整的隐私协议、用户授权流程 |
| **UGC 管理** | 证明有完善的举报和审核机制 |
| **免责声明** | 明确标注"本平台不提供医疗建议，内容仅供交流参考" |

## 十、风险清单与应对

| 风险 | 等级 | 应对策略 |
|------|------|----------|
| 虚假医疗信息传播 | 🔴 高 | 敏感词过滤 + 人工审核 + 用户举报三层防线 |
